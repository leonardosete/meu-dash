📄 Documentação Técnica - Dash Alertas

Este documento fornece uma visão detalhada da arquitetura, fluxo de dados e lógica de análise do projeto meu-dash.

1. Descrição do Projeto

O projeto Dash Alertas automatiza a análise de alertas de monitoramento (extraídos de arquivos .csv), gera um ecossistema de dashboards HTML interativos para gestão e identifica tendências de problemas ao longo do tempo.

O principal diferencial do sistema é a sua capacidade de classificar problemas com base em um Score de Prioridade Ponderado, que considera o risco para o negócio, a ineficiência da automação e o impacto operacional (volume de alertas), permitindo que as equipes foquem no que é mais crítico.

2. Lógica de Análise e Priorização

A análise se baseia em conceitos-chave para transformar "ruído" (milhares de alertas) em insights acionáveis (casos priorizados).

Casos vs. Alertas

Para focar na causa raiz, a análise distingue Alertas de Casos:

  • Alerta: Uma notificação individual de um evento de monitoramento.
  • Caso: A causa raiz de um problema. Ele agrupa múltiplos alertas do mesmo tipo que ocorrem em um mesmo recurso (CI/Nó), representando uma única frente de trabalho.
Exemplo: Um servidor com pouco espaço em disco pode gerar 100 alertas de "disco cheio". No entanto, como todos se referem à mesma causa raiz no mesmo recurso, eles são agrupados em apenas 1 Caso.

Score de Prioridade Ponderado

A criticidade de um caso é calculada pela fórmula:

Score Final = (Pilar 1: Risco) * (Pilar 2: Ineficiência) * (Pilar 3: Impacto)

Pilar 1: Score de Risco (Criticidade Base)

Mede a gravidade inerente do problema. É a soma dos pesos da Severidade e Prioridade do alerta, extraídos das colunas severity e sn_priority_group.

SeveridadePesoPrioridadePeso
Crítico10Urgente10
Alto / Major8Alto(a)8
Médio / Minor5Moderado(a)5
Aviso / Baixo2-3Baixo(a)2

Pilar 2: Multiplicador de Ineficiência (Falha da Automação)

Penaliza casos onde a automação de remediação falhou.

Resultado da AutomaçãoMultiplicador
Falha Persistente (nunca funcionou)x 1.5
Intermitente (falha às vezes)x 1.2
Status Ausente / Inconsistentex 1.1
Funcionou no final (Estabilizada / OK)x 1.0

Pilar 3: Multiplicador de Impacto (Volume de Alertas)

Penaliza o "ruído" operacional. O cálculo usa a fórmula 1 + ln(N) (logaritmo natural).

Contabilização da Remediação

A análise da remediação se baseia na coluna self_healing_status. O script analisar_alertas.py verifica o valor para cada alerta:

  • REM_OK: Sucesso.
  • REM_NOT_OK: Falha.
  • Ausente ou outro valor: Status desconhecido ou inválido.

A sequência desses status ao longo do tempo para um mesmo Caso define a Ação Sugerida (ex: "Falha Persistente", "Intermitente", "Estabilizada").

Qualidade dos Dados: Status de Remediação

Para garantir a integridade da análise, o script analisar_alertas.py realiza uma validação na coluna self_healing_status.

  • Valores Esperados: REM_OK, REM_NOT_OK, ou valores nulos/vazios.
  • Ação em caso de Inconsistência: Se um valor diferente dos esperados for encontrado, a linha inteira do alerta é registrada no arquivo invalid_self_healing_status.csv.
  • Notificação: O dashboard principal (resumo_geral.html) exibirá um aviso e um link para o qualidade_dados_remediacao.html, permitindo a fácil depuração e identificação de problemas na coleta de dados.

3. Arquitetura e Componentes

A arquitetura do projeto foi modernizada, migrando de um sistema baseado em scripts para uma aplicação web Flask que centraliza toda a lógica de orquestração e análise.

Orquestrador Principal: Aplicação Flask (src/app.py)

  • Responsabilidade: Único ponto de entrada e cérebro do projeto. Gerencia as interações do usuário, o fluxo de análise, a persistência de dados e o roteamento para servir os relatórios gerados.
  • Componentes Internos:
    • Flask: Microframework web que estrutura a aplicação.
    • SQLAlchemy: ORM para interagir com o banco de dados SQLite, gerenciando o histórico de relatórios.
    • Flask-Migrate: Ferramenta para gerenciar as migrações do esquema do banco de dados.
  • Rotas Principais:
    • GET /: Exibe a página inicial de upload (upload.html).
    • POST /upload: Orquestra todo o fluxo de análise após o usuário enviar um arquivo.
    • GET /relatorios: Exibe um histórico de todos os relatórios gerados.
    • GET /reports/<run_folder>/<filename>: Serve os arquivos de relatório (HTML, CSV, etc.).
    • GET /health: Endpoint de health check para Kubernetes. Retorna {'status': 'ok'}.
    • GET /docs/<path:filename>: Serve a documentação estática do projeto.

Banco de Dados (data/meu_dash.db)

  • Tecnologia: SQLite, gerenciado pelo SQLAlchemy.
  • Modelo de Dados (Report): Armazena metadados de cada análise. Os campos mais importantes são:
    • id, timestamp, original_filename.
    • report_path: Caminho para o resumo_geral.html.
    • json_summary_path: Caminho para o resumo_problemas.json, usado na próxima análise de tendência.
    • date_range: O intervalo de datas coberto pelo relatório (extraído do CSV).
    • max_date: Campo fundamental. Armazena a data mais recente do *conteúdo* do arquivo de input.
  • Responsabilidade: Manter um histórico de todas as análises. A comparação para análise de tendência agora é mais inteligente: em vez de apenas pegar a última análise executada, o sistema usa o campo max_date para encontrar a análise anterior mais relevante, cujo conteúdo seja cronologicamente anterior ao do upload atual.

Motores de Análise (Invocados como Bibliotecas)

src/analisar_alertas.py

  • Responsabilidade: Processar um único arquivo de alertas, realizar a análise principal e orquestrar a geração dos relatórios HTML para aquele período.
  • Invocação: É chamado como uma função Python (analisar_arquivo_csv) diretamente do app.py.
  • Entradas: Recebe o caminho do arquivo de input e o diretório de output como argumentos de função. A flag light_analysis controla a profundidade da análise.
  • Saídas: Retorna um dicionário contendo os caminhos para os artefatos gerados (HTML e JSON).

src/analise_tendencia.py

  • Responsabilidade: Comparar dois períodos (JSON atual vs. JSON anterior) e gerar o resumo_tendencia.html.
  • Invocação: É chamado como uma função Python (gerar_relatorio_tendencia) diretamente do app.py.
  • Entradas: Recebe os caminhos para os dois arquivos JSON (anterior e atual), os nomes dos arquivos CSV originais e os intervalos de datas como argumentos de função.

Módulos Auxiliares

src/gerador_html.py

Centraliza a lógica de renderização de HTML, transformando os dados da análise em dashboards interativos.

src/constants.py

Centraliza todas as constantes de lógica de negócio (pesos, ações, etc.) para garantir consistência.

src/get_date_range.py

Responsável por ler um arquivo de alertas e extrair a data mais antiga e a mais recente. Retorna uma tupla contendo o intervalo de datas formatado e o objeto de data da data mais recente (max_date).

4. Fluxo de Execução Lógico

O processo é iniciado pela interação do usuário com a interface web e orquestrado integralmente pelo app.py.

  1. Upload do Usuário: O usuário acessa a rota / e envia um arquivo .csv através do formulário, que faz um POST para a rota /upload.
  2. Orquestração (app.py):
    • a. Salvar e Extrair Datas: O .csv é salvo em data/uploads/. O módulo get_date_range.py é usado para extrair o intervalo de datas e a data máxima (max_date) do conteúdo do arquivo. Um diretório de saída (run_<timestamp>) é criado em data/reports/.
    • b. Buscar Histórico Relevante: O app consulta o banco de dados para encontrar o relatório anterior mais apropriado para comparação. A lógica é:
      1. Tenta encontrar o relatório cuja max_date seja a maior possível, mas ainda anterior à max_date do arquivo recém-enviado.
      2. Se nenhum relatório for encontrado com essa lógica (por exemplo, se for o primeiro upload ou se os dados forem mais antigos que todos os registros existentes), ele usa como fallback o último relatório inserido no banco, ordenado por timestamp.
  3. Análise de Tendência (Condicional): Se um relatório anterior relevante foi encontrado, o app.py:
    • i. Executa Análise Leve: Chama a função analisar_arquivo_csv com light_analysis=True para gerar rapidamente o resumo_problemas.json do arquivo atual.
    • ii. Gera Relatório de Tendência: Chama a função gerar_relatorio_tendencia, passando os JSONs (anterior e atual) e também os intervalos de datas (anterior e atual) para enriquecer o relatório.
  4. Análise Principal (Completa): A função analisar_arquivo_csv é chamada novamente, mas com light_analysis=False, para gerar o ecossistema completo de dashboards para o período atual. O link para o relatório de tendência é injetado no dashboard principal, se tiver sido gerado.
  5. Persistência e Redirecionamento:
    • i. Um novo registro Report é criado no banco de dados com os metadados da análise, incluindo os novos campos date_range e max_date.
    • ii. O app.py redireciona o navegador do usuário para a rota que serve o resumo_geral.html recém-criado.

5. Artefatos Gerados (Outputs)

Dashboards HTML Interativos

  • resumo_geral.html: Visão geral gerencial. Contém uma seção "Conceitos" que explica a metodologia da análise.
  • resumo_tendencia.html: (Modo comparativo) Relatório de evolução entre os períodos.
  • editor_atuacao.html: Editor interativo para o time atuar sobre os problemas que precisam de ação.
  • sucesso_automacao.html: Visualizador de casos resolvidos por automação.
  • instabilidade_cronica.html: Visualizador de casos que são remediados, mas ocorrem com alta frequência.
  • qualidade_dados_remediacao.html: (Condicional) Visualizador para os alertas com status de remediação inválido.
  • visualizador_json.html: Ferramenta de depuração para visualizar o resumo_problemas.json.
  • Outros: Planos de ação por squad, visualizador de squads, páginas de detalhes, etc.

Arquivos de Dados

  • resumo_problemas.json: A base de dados principal, com a análise completa do período.
  • atuar.csv: Lista de problemas que exigem ação manual.
  • remediados.csv: Lista de problemas resolvidos por automação.
  • remediados_frequentes.csv: Lista de problemas que indicam instabilidade crônica.
  • invalid_self_healing_status.csv: (Condicional) Log com os alertas que possuem um valor de status inesperado.
